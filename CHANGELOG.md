## [1.7.1] - 2026-05-16

Same-day v1.7.0 regression patch. Two distinct regressions surfaced in the first 12 hours after cutting v1.7.0, both caught by the production validation loop (Saturday canary cron + post-merge re-run of pr.yml on main). Single PR (#241) because both are tightly coupled to the v1.7.0 release cycle. **(1) Canary verify-testflight tag mismatch**: smoketest's Saturday cron failed at "Verify TestFlight ingestion" on both Local-mode cells (tuist + xcodegen) — `fastlane release` succeeded but `bin/verify-testflight.rb` polled ASC for tag `v1.7.0` instead of the canary's ephemeral CalVer tag, because the v1.7.0 release tag cut hours earlier on the smoketest defeated the git-tag-lookup heuristic. Fix: `verify-testflight.rb` now honors `RELEASE_TAG` env var override; `canary-local-mode.yml` passes the canary's computed tag through. **(2) AppStoreScreenshotTests cold-simulator flake**: `XCUIDevice.shared.appearance` setter timeout (~5-10% on cold GHA simulators) caught the post-merge re-run on main. Fix: switch to launch-argument-based color scheme override — `HelloAppMain` reads `-UITestColorScheme` and applies `.preferredColorScheme(.light/.dark)` at App init, bypassing the system appearance API entirely. Real users see no behavior change.

### Fixed

- **Critical fix (v1.7.0 regressions)**: two distinct regressions surfaced in the first 12 hours after cutting v1.7.0 — both caught by the actual production canary cron + post-merge re-run of pr.yml on `main`. **(1) Canary verify-testflight tag mismatch**: the Saturday 07:00 UTC canary cron failed at the "Verify TestFlight ingestion" step on BOTH the tuist and xcodegen Local-mode cells. `fastlane release` succeeded (smoketest's canary uploaded build `2102 (0.2026.20)` to ASC), but `bin/verify-testflight.rb` polled ASC for tag `v1.7.0` (marketing version `1.7.0`) and found no matches — because the v1.7.0 tag cut hours earlier on the smoketest was now the most recent local `v*` tag, and verify-testflight's `git tag --sort=-creatordate --list 'v*'` lookup returned it instead of the canary's ephemeral CalVer tag. The canary lane uses `skip_tag:true` so its computed `v0.YYYY.WW-canary-N-gen` tag is local-only and NEVER pushed; on any fork freshly cut from upstream, the most-recent pushed tag is the upstream release tag, defeating the git-based tag-lookup heuristic entirely. Fix: `bin/verify-testflight.rb` now honors `RELEASE_TAG` env var override before falling back to the git-tag lookup. `canary-local-mode.yml`'s "Verify TestFlight ingestion" step now sets `env: RELEASE_TAG: ${{ steps.ver.outputs.tag }}` so the canary's own CalVer tag is what verify polls against. Real-shipper use (`make verify` after `make ship`) is unchanged — `make ship` always pushes its tag so the git-lookup path still works. Surfaced 2026-05-16T07:35 UTC by the first Saturday cron after v1.7.0 was cut; smoketest run `25956612534` shows both Local-mode cells failing identically while both CI-mode cells succeeded (because CI-mode triggers `release.yml` which pushes its tag). **(2) AppStoreScreenshotTests `testLightMode` cold-simulator flake**: `pr.yml`'s `app (iOS Simulator)` (xcodegen) cell failed on the post-merge re-run of `release: cut v1.7.0 (#240)` on `main` with `Failed to set appearance mode: Timed out while setting appearance mode to Light`. `XCUIDevice.shared.appearance = appearance` waits for springboard confirmation; on cold-booted GHA simulators the confirmation handshake can timeout (~5-10% of runs). The canonical screenshot test pattern shipped in #216 used the API for both `testLightMode` + `testDarkMode`; `testLightMode` runs first alphabetically and gets the cold simulator. The pre-merge PR run had passed all 9 jobs (this fork of the test was warm by chance); the post-merge re-run on main caught the flake. Fix: switch from `XCUIDevice.shared.appearance = appearance` (system-API runtime setter, flaky on cold sims) to `app.launchArguments.append(["-UITestColorScheme", scheme])` (launch-time, deterministic). `HelloAppMain` (app/Shared/HelloApp.swift) reads `-UITestColorScheme` at App init and applies `.preferredColorScheme(.light)` or `.preferredColorScheme(.dark)` to its `WindowGroup`, bypassing the system appearance API entirely. Real users never pass `-UITestColorScheme`, so the system's own light/dark preference wins outside UI tests (zero behavioral change). Pattern: launch arguments → app reads → app renders. The XCUIDevice.Appearance enum is still used as the parameter type for `launchAndCapture` so the test call sites read identically (`testLightMode` still passes `.light`); only the internal mechanism changed. Validated end-to-end: HelloApp.swift compiles standalone with the new launch-arg reader; AppStoreScreenshotTests.swift still wires through the same `.light`/`.dark` choices. Both fixes ship as #241 because they're both v1.7.0 regressions in v1.7.0 features (release tag handling + canonical screenshot test pattern), found within hours of each other, and tightly coupled by the release cycle.

### Added

- `make doctor` step 16: `Xcode.app quarantine xattr`. Advisory-only check that surfaces a `com.apple.quarantine` xattr on `/Applications/Xcode.app` at minute zero of bootstrap, before any actual build/screenshots step. Top-level xattr scan only (~1ms — recursive scans take 30+ sec on the 12+ GB Xcode tree which is unacceptable for a doctor check); covers the canonical install vector (xcodes-cli / manual .xip downloads leave the bit on Xcode.app itself). Emits the actionable `sudo xattr -dr com.apple.quarantine /Applications/Xcode.app` one-liner and notes that `make screenshots` works around it automatically (so this is NOT blocking), but other dev-tool launches outside the template may hit Gatekeeper. No `do_it` runs — the fix needs sudo (Xcode.app is owned by root:wheel) and mutates a machine-wide app outside this fork's scope, so it stays user-controlled. Closes the discoverability gap surfaced by the 2026-05-11 screenshot Gatekeeper rejection: previously the user only learned about the quarantine bit after `make screenshots` failed ~15 minutes into onboarding; now `make doctor` flags it on first run.

- `make revoke-orphan-certs` — new Makefile target + `bin/revoke-orphan-certs.rb` driver + `revoke_orphan_certs` fastlane lane. Closes the recovery path for Apple's 3-cert distribution cap when CI's `if: always()` post-step has been skipped repeatedly (runner evictions, hard crashes, >7-day cache eviction of the orphan-tracking file) and the team has accumulated Apple-side distribution certs whose private keys live nowhere — neither on the user's Mac nor on any CI runner. Inverse direction of the existing `clean_revoked_certs` (which uses Apple's list as truth and deletes stale locals); this one uses your local Keychain as truth and revokes orphan Apple certs. Safety guard: matches by SHA-1 / serial against `~/Library/Keychains/login.keychain-db` — any cert with a matching local private key is treated as "in use" and never touched. Surfaced 2026-05-12 when `make ship` on prakashrj/my-cool-app hit `Could not create another Distribution certificate, reached the maximum number of available Distribution certificates` after canary cycles had left 3 orphan distribution certs on the shared `A26TJZ8QHQ` team. Previously the only recovery was the developer.apple.com web UI (~30 sec but error-prone — easy to revoke an in-use cert by mistake). Three flags: `DRY_RUN=1` (list orphans, no revoke), `YES=1` (skip confirmation prompt), default (interactive). Validated end-to-end on prakashrj/my-cool-app's live failure: 3 orphans correctly identified and revoked, 2 in-use distribution certs (the user's local-mode signing identities) correctly left alone, subsequent re-mint succeeded. Extended in #217 to also reap Development cert orphans (Apple caps Development certs independently from Distribution — ~2 per registered device vs 3 per team — with an identical failure-mode signature in the `cert` action): audit type set now covers both classes (`APPLE_DEVELOPMENT`, `DEVELOPMENT`, `IOS_DEVELOPMENT`, `MAC_APP_DEVELOPMENT` added alongside Distribution types); subject-string prefix list extended to match Development cert subjects (`"Apple Development"`, `"iPhone Developer"`, `"Mac Developer"`); safety guard unchanged.

- Canonical screenshot test pattern, borrowed and adapted from [buczel.com/blog/fastlane-screenshots-ios](https://buczel.com/blog/fastlane-screenshots-ios/). Five practices land in the template; four more documented as forker-side guidance in the new `docs/SCREENSHOTS.md`. Template-level changes:
  - New `app/Shared/AccessibilityIdentifiers.swift` defines the `AccessibilityIdentifiers` enum — single source of truth for UI test selectors. Compiled into BOTH the main app target (via `app/Shared/**`) AND the UI test targets (via explicit `sources:` entries in `project.yml` + `Project.swift`). UI tests run as a separate process and can't `@testable import` the app binary, so compiling the one shared file into both targets preserves the single-source-of-truth property without requiring testability flags. Starts with one identifier (`title = "HelloApp.title"`) — minimal but extensible; the doc-comment explains the naming convention forks should follow.
  - `app/Shared/ContentView.swift` attaches `.accessibilityIdentifier(AccessibilityIdentifiers.title)` to the title Text. (The pre-existing `.accessibilityIdentifier("HelloApp.stub")` on the VStack was removed — SwiftUI containers without an explicit accessibility role don't surface in XCUITest queries, so the identifier was effectively dead code.)
  - `app/UITests/AppStoreScreenshotTests.swift` rewritten with per-appearance test functions (`testLightMode` + `testDarkMode`) instead of a single golden-path function. Fastlane sets `XCUIDevice.shared.appearance` BEFORE `app.launch()` so the app boots directly in light or dark mode — no relaunch, no flicker. Pattern wins: parallelizable across simulators, independent failure isolation (dark breaking doesn't lose light captures), independent re-run (`--only_testing AppStoreScreenshotTests/testLightMode` when iterating). Adds `XCUIDevice.shared.orientation = .portrait` in `setUpWithError()` — forces portrait every run; Simulator orientation persists across runs and ASC rejects mis-rotated uploads (iPhone 16 Plus requires 1290×2796 portrait; a landscape capture produces 2796×1290 = same pixels, wrong dimensions, rejected). Wait target switched from `app.staticTexts["HelloApp"]` (visible-text match — fragile to localization) to `app.staticTexts[AccessibilityIdentifiers.title]` (identifier match — locale-proof, refactor-safe).
  - `fastlane/Snapfile` flips `override_status_bar(false)` → `override_status_bar(true)`. Pins simulator status bar to 9:41 / full battery / full wifi for clean App Store screenshots (Apple's own marketing convention).
  - New `docs/SCREENSHOTS.md` documents what the template ships + forker-side practices that go beyond the template's "one screen × one language × two appearances" canonical scaffolding: mock data based on real API responses (live servers in UI tests are flaky), splash screen skip via `-UITest` launch argument, "test one config first, verify by listing folder contents" discipline, identifier placement (which SwiftUI elements actually surface in XCUITest queries), iOS 26 TabView identifier quirks (template doesn't use TabView yet but forks adding tab navigation will hit it).
  - Validated end-to-end on apple-shipkit's local tree: `make screenshots` exits 0, all four iOS captures produced (iPhone 16 Plus + iPad Pro 12.9" × light + dark), PNG dimensions exactly match App Store Connect requirements (1290×2796 + 2048×2732), status bar shows 9:41 / full battery / full wifi, light vs dark cleanly differentiated.

- `fastlane/Fastfile.local` extension point. The template's `Fastfile` now imports `fastlane/Fastfile.local` if present at the bottom of the file (after all lane definitions), giving forks a clean place to add custom lanes, `after_all` / `error` hooks, and `override_lane` rewrites without editing the template. Closes the fastlane-extension regression class: previously, every upstream `Fastfile` change (this session alone touched it 12+ times) would conflict with a fork's edits and require manual resolution. Now `Fastfile.local` is fork-owned and survives `git pull upstream main` with zero touches. New `fastlane/Fastfile.local.example` ships ~7 copy-pasteable patterns adapted from [`fastlane/examples`](https://github.com/fastlane/examples) — Slack/Discord `after_all`+`error` hooks, `lane :ship_with_crashlytics` (wraps the template's `release` with `download_dsyms`+`upload_symbols_to_crashlytics`), `lane :beta` for Firebase App Distribution (ad-hoc profile + non-TestFlight push), App-Store-submission Linear/Jira ticket bumps, custom action skeleton for fastlane/actions/, internal version-tracking pings. All template constants (`APP_NAME`, `APP_IDENTIFIER`, `IOS_SCHEME`, `MACOS_SCHEME`) and helpers (`asc_api_key`, `pilot_with_retry`, `changelog_text`, `extract_changelog_section`, `parse_release_tag`, `cert_options_for_type`) are reachable from `Fastfile.local` because fastlane composes imported files into a single namespace. Hook conventions documented inline: template claims `before_all` for ASC setup (forks must NOT override — last-write-wins would clobber the ASC bootstrap); `after_all` and `error` are free for forks; bottom-of-file import is intentional so `override_lane` works. New `docs/MAINTAINING-A-FORK.md` section "I want to add custom lanes" walks forkers through the cp-and-uncomment workflow with the full pattern table. Validated end-to-end: smoke test confirmed `Fastfile.local` registers a forker lane (`fastlane lanes` lists it), the imported file resolves template constants (`APP_NAME=HelloApp` etc.) and template helpers (`parse_release_tag('v1.2.3+4')` returns `["1.2.3", "4"]`), and the absent-`Fastfile.local` path is a clean no-op (file existence-checked before `import`).

- `make doctor` step 17: `Leaked fastlane temp keychains`. Advisory-only check that detects `fastlane_tmp_keychain*` entries in the user keychain search list. Surfaced 2026-05-12 when a forker (the maintainer dogfooding) hit recurring Google Drive auth-prompt popups: `"Google Drive wants to use the 'fastlane_tmp_keychain-db' keychain. Please enter the keychain password."` Diagnosis: `setup_ci` (called from `before_all` in CI mode) creates `~/Library/Keychains/fastlane_tmp_keychain-db` at fastlane start and registers an `at_exit` cleanup. If the fastlane process is killed (Ctrl-C, SIGKILL, OOM, crash, runner-eviction-mid-flow) before `at_exit` fires, the keychain leaks into the user's keychain search list and any app that walks all keychains for credentials (Google Drive, 1Password, iCloud Keychain, browser auth dialogs) repeatedly prompts for its password. Detection via `security list-keychains -d user` + regex match on `fastlane_tmp_keychain` substring (~1ms). Linux runners short-circuit clean (no Keychain Services). Cleanup advisory ships as a TWO-step recipe (delete + restore-default; see #238 for why) so following our own advice doesn't break `make ship` afterward. No `do_it` runs — `security delete-keychain` is destructive (removes from search list AND deletes file); leaked keychains typically hold no real state, but mutating user-keychain state stays user-controlled, mirroring the XcodeQuarantine policy from #211.

- `make adopt` + `bin/adopt.rb` + `fastlane/Fastfile`'s new `adopt_existing_app` lane + `docs/ADOPTING-EXISTING-APP.md` walkthrough — the existing-app migration path. Closes the catastrophic "fork the template, run `make submit`, clobber the live App Store listing with template placeholders within minutes" failure mode that's been latent since v1.0. Single command pulls existing ASC state down to disk before first ship: verifies the ASC App record exists for the configured `BUNDLE_ID` (fail-loud with actionable next steps if not — checks team id, API key validity, bundle id correctness), then `bundle exec fastlane deliver download_metadata --force --api_key_path JSON --app_identifier ID --metadata_path PATH` for all locales of metadata (description, keywords, marketing/support/privacy URLs, name, subtitle, promotional text, release notes), then `bundle exec fastlane deliver download_screenshots --api_key_path JSON --app_identifier ID --screenshots_path PATH` for all device classes × all locales of screenshots, then reports the latest ASC marketing version + state for the user to manually sync into `app/project.yml` (xcodegen) or `app/Project.swift` (tuist) — deliberately NOT auto-edited because the user might be doing a deliberate version bump simultaneously, or might use a custom format (`3.4.2-rc1`) the substitution would mangle. Idempotent: re-runs re-sync from ASC. `bin/adopt.rb` driver does preflight (refuses to run if `BUNDLE_ID=com.example.helloapp` template placeholder; refuses to run if uncommitted changes in `fastlane/metadata` or `fastlane/screenshots` would be lost — `FORCE=true` overrides). `fastlane/Fastfile`'s lane does its own placeholder check (defensive — catches the case where someone bypasses `make adopt` and invokes `fastlane adopt_existing_app` directly with full ASC creds). Skip flags: `make adopt SKIP_METADATA=true` (screenshots only) or `make adopt SKIP_SCREENSHOTS=true` (metadata only, ~5 sec). `bin/lib/bootstrap.rb`'s existing `ScanMetadata` step (doctor step 14) now appends a contextual hint to its placeholder warning when `BUNDLE_ID` is non-placeholder AND `ASC_API_KEY_ID` is set — surfaces "If your fork is adopting a LIVE App Store app, do NOT just edit these files — running `make submit` would overwrite your real App Store listing. Run `make adopt` first." Apple identifies apps by `Bundle ID + Team ID` only, so the template-shipped certs (mint-fresh per CI release run) are interchangeable with whatever shipped previous builds — TestFlight build history continues uninterrupted, App Store ratings + reviews + iCloud data + Keychain entries + IAP records all persist transparently for end users. AGENTS.md gains a new "Adopting an existing live App Store app" section flagging this as a hard requirement (MUST run `make adopt` before `make submit` for existing-app forks). README's five-command-journey section adds an existing-app callout pointing to the new doc.

- `make doctor` step 18: `Default keychain set`. Advisory-only check that probes `security default-keychain -d user` and emits `[:warn, msg]` with the actionable one-liner (`security default-keychain -d user -s ~/Library/Keychains/login.keychain-db`) when no default is set. fastlane's sigh (called by `make ship`) decodes provisioning profiles via `security cms -D -i <profile>` internally — that decode REQUIRES a default keychain. Without one, sigh fails with `Failure to decode <profile>. Exit: 1: security: cert import failed: A default keychain could not be found.` — `make ship` aborts before any signing happens. Most common cause: a prior `security delete-keychain` (e.g. cleaning up a leaked fastlane temp keychain via the FastlaneTmpKeychain advisory) removed the keychain that was set as default, since fastlane's `setup_ci` action sets its temp keychain as user-default via `default_keychain: true`. Apps that walk the search list (Keychain Access, Wi-Fi auth, browser saved passwords) keep working silently — only tools that probe `SecKeychainCopyDomainDefault` directly notice. No `do_it` — mutating user-keychain state stays user-controlled per the XcodeQuarantine + FastlaneTmpKeychain policy; auto-setting login.keychain-db could fail confusingly if missing/corrupted, or silently override intentional hardened-mac configurations. Pattern continuity (#211, #229, this): advisory-only doctor steps for state outside the template's scope.

- **Release tag format → `v<MARKETING>+<BUILD>` (e.g. `v1.0.0+5`)**, replacing the previous CalVer-as-marketing-version scheme (`v2026.19.1357`, `v2026.19.1`). Result: **one TestFlight "Version 1.0.0" bucket holding all builds shipped at marketing 1.0.0**; bumping marketing version is intentional (= "this release introduces real changes worth telling users about"). New `bin/lib/version_resolver.rb` (Bootstrap::Version) + `bin/compute-release-tag.rb` CLI used by both local-mode `bin/ship.rb` and CI-mode `release.yml`'s "Compute release tag" step. ## Versioning

This CHANGELOG tracks **template versions** — releases of the `apple-shipkit` repo itself, not the forker apps that consume it. Forker apps maintain their own CHANGELOG and version policy independently.

Pre-1.0 (`0.x`) is the development phase: breaking changes are allowed without bumping a major version. The `0.x` retrospective entry above summarizes what shipped during M1–M4 milestones; further pre-1.0 work lands under `[Unreleased]` until the public flip.

`v1.0.0` was cut at the public-visibility flip (2026-05-01). From `v1.0.0` onward this project follows strict [Semantic Versioning](https://semver.org/spec/v2.0.0.html): `MAJOR.MINOR.PATCH` where MAJOR is incremented for breaking forker-facing changes (renamed `bin/` scripts, removed Make targets, restructured README sections that forkers cite), MINOR for new forker-facing features (new scripts, new sections), and PATCH for fixes that don't change the forker contract.

Forkers tracking template upgrades should read the CHANGELOG before pulling template changes into their fork. Template-fork creation always uses the default branch (`main`); to base a fork on a specific template version, use `git fetch` to pull a specific tag and rebase manually.

## Tagging

When cutting a release:

1. Move the `[Unreleased]` entries into a new `[X.Y.Z] - YYYY-MM-DD` section in `CHANGELOG.md` and commit.
2. Extract the new section to a release-notes file: `awk '/^## \[X\.Y\.Z\]/{flag=1; print; next} flag && /^## /{exit} flag' CHANGELOG.md > release-notes.md`.
3. Create an annotated tag with the release notes as its message, push, and create the GitHub release.

```bash
# After moving Unreleased → [X.Y.Z] in CHANGELOG.md, committing,
# and writing release-notes.md (the new section's body):
git tag -a vX.Y.Z --cleanup=verbatim -F release-notes.md   # annotated tag; --cleanup=verbatim preserves Markdown headings (default --cleanup=strip eats #-prefixed lines); -F reads notes from file
git push origin vX.Y.Z
gh release create vX.Y.Z --notes-from-tag --title "vX.Y.Z" --verify-tag
```
