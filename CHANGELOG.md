## [Unreleased]
### Added

- `.github/dependabot.yml` ‚Äî weekly bundler + GitHub Actions dependency updates. Bundler ecosystem is grouped (fastlane/spaceship gems batched into one PR; everything else into another) so reviewers see one consolidated diff per week instead of N separate gem-bump PRs. github-actions ecosystem groups by author family (`actions/*`, `ruby/*`). The Gemfile pins `fastlane ~> 2.224` (per docs/CONTINUOUS-VALIDATION.md G11) so dependabot will only PR compatible bumps within that range, preserving the SHA-pin invariant + the G11 workaround. Forkers inherit the config; they can lower the cadence or add ignores without touching anything else.
- `.ruby-version` (`3.3`) + `.tool-versions` ‚Äî pins Ruby + xcodegen + tuist + lefthook + swiftlint + xcbeautify + swiftformat for forkers using asdf / mise / rtx. Other forkers ignore the file (Brewfile remains the canonical source for `make bootstrap`). Eliminates the "works on my machine" failure mode where a forker's xcodegen 2.30 misparses the project.yml that the maintainer's xcodegen 2.45 wrote, and similar drift across the toolchain.
- `.editorconfig` ‚Äî universal editor settings for VSCode, Cursor, JetBrains, Vim, Emacs, Sublime. Per-language indent rules (Swift 4-space, Ruby/YAML/JSON 2-space, Bash 2-space, Makefile MUST be tab); LF line endings everywhere except .bat; trim trailing whitespace except in Markdown (where two-space-trailing = line break per CommonMark) and in plist/pbxproj/xcconfig/xcodeproj (Xcode formats those in specific ways). Without this, multi-language template forks accumulate tab-vs-space diffs across the codebase as different contributors' editors apply different defaults.
- `.swiftformat` config + `swiftformat` Brewfile entry + `make format` / `make format-check` Makefile targets + `swiftformat` job in `.github/workflows/pr.yml`. Companion to swiftlint: swiftlint flags style violations, swiftformat auto-fixes them. Excluded paths match swiftlint's (Tuist Derived auto-generated, fastlane SnapshotHelper "do-not-modify", app/Project.swift Tuist DSL whose conventions fight default SwiftFormat rules). Conservative defaults ‚Äî `--swiftversion 5.9`, `--indent 4`, `--commas inline`, `--self remove`. PR check runs `swiftformat --lint app/` and fails on any diff. Forkers can downgrade to a warning by editing the job, or disable swiftformat entirely by removing the job + the Brewfile line + the `.swiftformat` file.
- `fastlane/Fastfile` `submit_for_review` lanes auto-create a published GitHub Release for the marketing version being submitted. Per industry standard for app teams (Slack, 1Password, Notion-iOS), GitHub Releases mark *App Store submissions* ‚Äî not every TestFlight build. Per-build traceability is preserved by the per-ship `vMARKETING+BUILD` git tags (still pushed by the `release` lane); this helper opportunistically upgrades the most-recent of those tags into a published Release whenever `submit_for_review` runs. Tag points at the latest `vMARKETING+*` tag (the actual build being submitted), not HEAD ‚Äî clicking the Release shows the exact commit Apple is reviewing. Notes pulled from `CHANGELOG.md`'s matching `## [<MARKETING>]` block via the existing `extract_changelog_section` helper. Idempotent across iOS + macOS submissions of the same marketing version: the second invocation sees the existing Release and no-ops. Skip-on-failure rather than fail-the-lane (App Store submission already succeeded by the time this runs; a Release-creation flake shouldn't undo it). Disable per-invocation via `RELEASE_SKIP_GH_RELEASE=true`. Skips silently when `gh` CLI isn't installed/authenticated, when no matching tag exists locally, or when the Release already exists.

- `make submit` target ‚Äî PLATFORMS-aware, opt-in auto-submit. Stages metadata + screenshots + build attachment + export-compliance for App Store review across the platforms in `.bootstrap.env`; `SUBMIT_FOR_REVIEW=true` (default `false`) flips from stage-only to auto-submit. Closes the asymmetry where `make ship` read PLATFORMS but submission required two manual `bundle exec fastlane <platform> submit_for_review` invocations. New `bin/submit.rb` driver mirrors `bin/ship.rb`'s shape (preflight summary, per-platform dispatch, env > config override). New Fastfile lanes `:stage_for_review` (iOS + macOS) call the existing `do_submit_for_review` helper with `submit: false`; the existing `:submit_for_review` lanes are unchanged (still auto-submit) so direct fastlane CLI invocations preserve their behavior. GH Release side-effect (#175) is now gated on the real-submit path ‚Äî staging is reversible, creating a "submitted" Release for a version that may never get submitted would be misleading. Pre-flight gates: PLATFORMS resolves to ‚â•1 platform, `fastlane/screenshots/en-US/` has ‚â•1 PNG/JPG matching the per-platform glob (Mac* for macOS, [!M]* for iOS), `fastlane/metadata/en-US/` exists. Per-invocation overrides via env vars (`SUBMIT_FOR_REVIEW=‚Ä¶`, `PLATFORMS=‚Ä¶`, `RELEASE_SKIP_GH_RELEASE=‚Ä¶`) ‚Äî same pattern as `make ship`. New `.bootstrap.env.example` "Submission behavior" section explains the maturity ramp. Maintainer doc updated in `docs/MAINTAINING-A-FORK.md` with stage-vs-submit table + recommended progression (start `false`, flip to `true` after 3-4 successful releases).

### Fixed

- `ci/take-screenshots.sh` + `ci/bump-asc-version.sh` ‚Äî Ruby version pin reads `.ruby-version` instead of using the unversioned brew `ruby` symlink. brew bumped `/opt/homebrew/opt/ruby` from 3.3 to 4.0 in the May 2026 cycle; that symlink is what both scripts had hardcoded for years (under the legitimate "avoid system Ruby" rationale). Bundler under Ruby 4.0 looks for gems at `vendor/bundle/ruby/4.0.0/`; `make bootstrap` had installed them at `vendor/bundle/ruby/3.3.0/` per `.ruby-version` ‚Äî instant `Could not find fastlane-2.230.0, CFPropertyList-3.0.8, abbrev-0.1.2, ‚Ä¶` for every gem in the lockfile. Surfaced via prakashrj/my-cool-app `make screenshots` after #181 unblocked the bundle-prereq message. Now: read `.ruby-version` (defaulting to `3.3` if absent), extract MAJOR.MINOR, prefer `/opt/homebrew/opt/ruby@${MAJOR.MINOR}/bin` if installed, fall back to unversioned `/opt/homebrew/opt/ruby/bin` with a loud WARN line so the failure mode is observable not silent. Also added `brew "ruby@3.3"` to `Brewfile` so `make bootstrap` actually installs the pinned Ruby (was missing ‚Äî forkers relied on whatever Ruby happened to be present from prior installs).
- `ci/take-screenshots.sh` ‚Äî APP_NAME auto-detection from generated `.xcodeproj` basename. Was relying on `bin/rename.sh` to substitute `HelloApp` ‚Üí `<APP_NAME>` in the `-project app/HelloApp.xcodeproj`, `-scheme HelloApp-macOS`, `-only-testing:HelloAppMacOSUITests/...` literals. Worked for forkers who ran `bin/rename.sh` once and never touched the script again, but anyone who copied a fresh upstream version of the script over their fork's substituted version (e.g. a maintainer mirroring an upstream fix) would re-import HelloApp references and break their fork. Now: glob `app/*.xcodeproj` after generate, basename ‚Üí APP_NAME, derive scheme + test-target names from APP_NAME via shell expansion. No template substitution needed in this script anymore (cosmetic refactor ‚Äî the substitution bin/rename.sh does still works, just produces dead literals that aren't referenced). Self-contained and forker-fork-safe.
- `Makefile` ‚Äî `screenshots:` target now has `_check-bundle` prereq alongside every other fastlane-invoking target (`ship`, `verify`, `submit`, `release-dryrun`, `doctor`, `bootstrap-fork`, `mint-local-certs`, `clean-revoked-certs`). Was the only outlier ‚Äî running `make screenshots` on a fresh fork before `bundle install` produced a giant unhelpful `Could not find fastlane-2.230.0, CFPropertyList-3.0.8, abbrev-0.1.2, ‚Ä¶` gem-list dump from bundler instead of the friendly `Ruby gems aren't installed yet. Run one of: bundle install / make bootstrap` message every other target prints. Surfaced via prakashrj/my-cool-app `make screenshots` after #180 unblocked the generator step. One-line fix; all forkers benefit on next `make screenshots`.
- `ci/take-screenshots.sh` ‚Äî generator-aware. Was hardcoded to `xcodegen generate` on line 47, which broke `make screenshots` for any Tuist-only fork (`bin/switch-to-tuist.sh` deletes `app/project.yml` so xcodegen has nothing to read). Surfaced via prakashrj/my-cool-app's first attempt at `make screenshots` after #179: `No project spec found at app/project.yml`. Now mirrors the matrix-builder pattern in `.github/workflows/pr.yml`: detect filesystem presence (`app/project.yml` ‚Üí xcodegen, `app/Project.swift` ‚Üí tuist; xcodegen wins if both are present), invoke the appropriate generator with `--no-open` for tuist. Fails loudly with a remediation message ("Run bin/switch-to-{tuist,xcodegen}.sh") when neither manifest is found. App-name substitution in this script (HelloApp ‚Üí SmokeApp) was already handled by `bin/rename.sh`; only the GENERATOR axis needed runtime detection because forkers can flip generators independently of renaming.

- `bin/setup-github.sh` ‚Äî adds `swiftformat` to the always-on `CHECKS` array alongside `swiftlint`. Tier 1 (#175) added the `swiftformat` *job* to `pr.yml` and patched branch protection on apple-shipkit + smoketest by hand (`gh api PATCH`), but the install path that new forkers hit (`make bootstrap-fork` ‚Üí `bin/setup-github.sh`) was never updated. Without this, a fresh fork's `swiftformat` job runs on every PR but is **not** required for merge ‚Äî a red swiftformat could ship to main. Required-checks count goes from 7 to 8 (or 5 to 6 in single-platform forks).
- `README.md` + `docs/PRINCIPLES.md` + `docs/BOOTSTRAP.md` ‚Äî sync the "7 required checks" claim across all three doc surfaces to "8". `README.md`'s tree comment for `Brewfile` adds swiftlint + swiftformat (was: just lefthook + xcodegen + tuist + fastlane). `Makefile` tree comment adds `format` / `format-check`. The "Ship a new version" cell in the README also drops the stale `CalVer: v2026.20.<run-number>` example ‚Äî the current scheme is `v<MARKETING>+<BUILD>` from #172.
- `docs/MAINTAINING-A-FORK.md` ‚Äî disclose `submit_for_review`'s new GitHub Release side-effect from #175 (idempotent across iOS+macOS, notes from `CHANGELOG.md`, opt-out via `RELEASE_SKIP_GH_RELEASE=true`). Without this paragraph, a forker's first App Store submission would surprise them with an unexpected GitHub Release; the env-var opt-out also wasn't documented anywhere user-facing (only in a Fastfile comment).

### Changed

- **Release tag format ‚Üí `v<MARKETING>+<BUILD>` (e.g. `v1.0.0+5`)**, replacing the previous CalVer-as-marketing-version scheme (`v2026.19.1357`, `v2026.19.1`). The old scheme conflated "which run shipped this" with "what marketing version users see" ‚Äî every ship invented a new `CFBundleShortVersionString`, fragmenting TestFlight into one "Version X" bucket per ship (the user-visible symptom: 6 buckets after 6 ships, App Store submission flow ambiguous, testers scrolling past dozens of obsolete entries). Industry-standard pattern (used by Slack, Spotify, Instagram, every shipped iOS app): marketing version is human-meaningful and stable across many ships; build number is the per-upload unique identifier ASC requires. Now: marketing version is read from `app/project.yml`'s `MARKETING_VERSION` (xcodegen) or `app/Project.swift`'s `"MARKETING_VERSION"` (tuist) ‚Äî bumped intentionally by editing the project file when you cut a real release. Build number is resolved from ASC at ship time as `max(existing builds for this marketing version) + 1`, threaded through the tag's `+N` suffix, then unpacked by `ci/local-release-check.sh` into `xcodebuild MARKETING_VERSION=‚Ä¶ CURRENT_PROJECT_VERSION=‚Ä¶`. Result: **one TestFlight "Version 1.0.0" bucket holding all builds shipped at marketing 1.0.0**; bumping marketing version is intentional (= "this release introduces real changes worth telling users about"). New `bin/lib/version_resolver.rb` (Bootstrap::Version) + `bin/compute-release-tag.rb` CLI used by both local-mode `bin/ship.rb` and CI-mode `release.yml`'s "Compute release tag" step. Backwards-compatible: `verify-testflight.rb` parses both new (`+`-separator) and old (no separator or `-`-separator) tag formats; `ci/local-release-check.sh` parses all three formats including the canary's `v0.YYYY.WW-canary-N-gen`. Override hooks preserved: `RELEASE_MARKETING_VERSION` (canary uses this to pin a per-week marketing version on a shared bundle without editing project.yml), `RELEASE_BUILD_NUMBER` (CI used to thread `github.run_number`; now obsolete since the tag carries the build number, but still honored).
- `bin/verify-testflight.rb` ‚Äî when the tag carries a build number (new format), reports per-build pinpoint match (` ‚Üê` next to the matching build) so the user can see which of the listed builds is THEIRS (when the smoketest's canary or another shipper is uploading concurrently to the same bundle). Falls back to per-platform listing without pinpoint mark for legacy tags.
- `.github/workflows/release.yml` ‚Äî "Compute release version" step now calls `bundle exec ruby bin/compute-release-tag.rb` instead of synthesizing `v$YYYY.$WW.$run_number`. Drops the `RELEASE_BUILD_NUMBER: github.run_number` env on the fastlane release step (build number is now carried by the tag's `+N` suffix). `workflow_dispatch.inputs.version` description updated to reflect the new format. Manual override path preserved ‚Äî pass an explicit tag like `v1.0.0+1` to skip the resolver entirely.
- `bin/ship.rb` local-mode ‚Äî replaces `tag = "v#{Time.now.utc.strftime('%Y.%V.%H%M')}"` with `Bootstrap::Version.compute_release_tag(config["BUNDLE_ID"])`. Requires ASC credentials at ship time (already required for the upload step itself; no new dependency). Falls back to a clear error message if the project file lacks a `MARKETING_VERSION` field or ASC is unreachable.

### Added

- `bin/ship.rb` pre-flight summary ‚Äî prints the ref, app name, bundle id, mode, signing path, and platforms before triggering release.yml or running `fastlane release` locally. Surfaces config drift early (e.g. `APP_NAME=SmokeApp` on a repo named `my-cool-app` stands out at a glance, instead of getting silently shipped). CI-mode preflight also clarifies that GitHub Actions runs a `PR` *workflow* (file: `.github/workflows/pr.yml`) in parallel on every push ‚Äî that's a CI sanity check, **not** a Pull Request, and it does **not** gate the release. Without the note, first-time forkers were reading `PR #2: failed` in the Actions UI and assuming an actual PR hadn't merged yet, when in fact `make bootstrap-fork` pushes the rename directly to main (no PR opened) and the "PR" label is just GitHub's display of the workflow's run number.
- `bin/lib/bootstrap.rb` end-of-pipeline summary ‚Äî extends the `‚úÖ Bootstrap complete` block with a "what just happened" recap (rename committed + pushed direct to main, no PR opened) and, in CI mode, the same parallel-pr.yml clarification. Closes the documentation gap between `make bootstrap-fork` and `make ship` where a fresh forker had no signal that the "PR #N: failed" notification on GitHub was orthogonal to the ship.

### Fixed

- `.github/workflows/pr.yml` matrix builder + `bin/setup-github.sh` required-checks list ‚Äî both now detect which generator manifests are committed (`app/project.yml` ‚Üî xcodegen, `app/Project.swift` ‚Üî tuist) and only emit/require what actually runs. Single-generator forks (the typical end state ‚Äî pick xcodegen OR tuist via `bin/switch-to-{xcodegen,tuist}.sh`) had two related breakages stacked: (1) pr.yml's matrix hardcoded `for gen in xcodegen tuist` so the 3 cells for the unselected generator failed on every push (`xcodegen|tuist generate` had no manifest to read), and (2) setup-github's branch protection required all 6 check names so even after fixing pr.yml, PRs would show "N of 7 required checks expected" and never become mergeable ‚Äî the 3 absent-generator names would never appear. Now both read filesystem presence and emit/require only matching cells. apple-shipkit + smoketest (both manifests committed) keep running both as before. Surfaced via prakashrj/my-cool-app freshly bootstrapped with `GENERATOR=tuist`: release.yml shipped successfully on commit `dff5178` (run `25617982936`, tag `v2026.19.1`), but the same commit's pr.yml run `25617977534` had 3 red xcodegen cells alongside 3 green Tuist ones, and branch protection's hardcoded 7-check list blocked all PR merges until the contexts were PATCHed live to drop xcodegen names. Existing forks should re-run `bin/setup-github.sh` to apply the corrected branch-protection contexts. Cleanup: `bin/switch-to-{tuist,xcodegen}.sh` had `mutate_pr_workflow` blocks running sed against text patterns (`name: install xcbeautify + xcodegen`, `run: brew install xcbeautify xcodegen`) that no longer existed in pr.yml after the matrix refactor ‚Äî dead code, removed. Mutation count: tuist 5‚Üí4 surfaces, xcodegen 6‚Üí5.
- `bin/verify-testflight.rb` ‚Äî `make verify` was reporting `Build.all(sort: -uploadedDate, limit: 5).first` as "the latest build", which on any fork sharing its Bundle ID with another shipper (e.g. a fork using the same ASC App as the smoketest's canary, or a team sharing one Apple App across multiple devs) surfaced whatever was last uploaded by anyone ‚Äî not the build the human just shipped via `make ship`. Now derives the expected marketing version from the latest local `v*` git tag (`tag.sub(/^v/, "").split("-", 2).first`), server-side filters `Build.all(version: ‚Ä¶)` to that exact version, and reports per-platform state (catches partial uploads where iOS shipped but macOS failed). Falls back to old "newest by time" behavior with a loud warning when no local `v*` tag exists (fresh fork that hasn't shipped yet). (#147)
- `.github/workflows/release.yml` ‚Äî full rewrite. Replaces the match-based CI signing path with the canary-local-mode pattern: each run mints fresh G6-issued certs (Apple Distribution + Apple Development + 3rd Party Mac Developer Installer) into a controlled keychain, ships, then revokes the just-minted ASC ids on `always()`. Net cert delta per run = 0. Workflow_dispatch only (no cron); single cell (no matrix.generator); `skip_tag:false` (real release tag pushed to origin). New `changelog` workflow input feeds `RELEASE_CHANGELOG` to the lane's annotate step (auto-extracts from `CHANGELOG.md` if input empty). Drops `MATCH_PASSWORD`, `MATCH_GIT_BASIC_AUTHORIZATION`, `RELEASE_MODE` env vars + the WWDR pre-install step (fresh G6 certs don't need legacy WWDR generations). Validated end-to-end on `prakashrj/my-cool-app` ship: `v2026.19.10` tagged + uploaded + annotated successfully. (#158)
- `fastlane/Fastfile` `release` lane ‚Äî drops the `if release_mode == "ci"` match branch entirely. The lane is now always sigh-based: `sigh` fetches App Store profiles via ASC API, cert SHA-1 extracted from profile `DeveloperCertificates[0]`, xcodebuild pinned to that SHA-1 via `CODE_SIGN_IDENTITY`. Works identically in local mode (login keychain, user's certs auto-minted by `make bootstrap-fork` / `make mint-local-certs`) and CI mode (controlled release keychain, certs minted by `release.yml`'s `mint_canary_certs` step before the lane runs). `RELEASE_MODE` config field stays as a knob for `bin/ship.rb` (run lane locally vs trigger CI workflow), but no longer differentiates signing path. (#158)
- `docs/CONTINUOUS-VALIDATION.md` G16 entry ‚Äî documents the macos-15-arm64-20260427 runner image regression that broke legacy WWDR-G3-issued certs at exportArchive even with full chain (G2-G6) pre-installed in System + login keychains. The mint-fresh-per-run approach (#158) sidesteps the issue by never reusing legacy-issued certs. Surfaced today via fresh CI ship attempt on prakashrj/my-cool-app reusing indiagrams/ios-macos-smoketest-certs (G5HJYWLM9Z, issued by WWDR G3, exp 2027-05-06).
- `fastlane/Fastfile` `release` lane ‚Äî populates the just-uploaded build's `BetaBuildLocalization[en-US].whatsNew` (TestFlight "What to Test") for ALL shippers ‚Äî canary + human local-mode + CI mode ‚Äî instead of only the canary's standalone post-upload step. Before this PR, every `make ship` produced a build with empty "What to Test" because fastlane pilot 2.233.1's `set_changelog` is structurally broken for new builds with no pre-existing BBL (G15). Three new helpers landed in `fastlane/Fastfile`: `changelog_text` (resolves text from `:changelog` opt ‚Üí `RELEASE_CHANGELOG` env ‚Üí CHANGELOG.md `## [<version>]` first non-empty block ‚Üí `Build #{tag}` last-resort), `ascii_sanitize` (replaces typographic non-ASCII with ASCII fallbacks; ASC's BBL endpoint rejects non-Latin in `whatsNew`), and `annotate_testflight_whats_new` (Spaceship POST/PATCH with poll-for-ASC-indexing loop, mirrors the canary's empirical 10-min ceiling). New release-lane opts `changelog:` and `skip_changelog:`. Canary's standalone Annotate step removed (lane handles it). Verified live by backfilling existing my-cool-app builds via direct probe. (#149)

## [1.5.0] - 2026-05-09

Local-mode shipping path canary (#125). Continuously validates the path stabilized in v1.4 (#115/#117/#119/#121/#123) so regressions surface within a week instead of at the next release. One-time setup (revoke 1 spare DIST + 1 spare MAC_INSTALLER cert per team, ~30 sec via portal or `fastlane revoke_certs ids:‚Ä¶`) dedicates 1 cycling slot per at-cap type ‚Äî empirically verified caps via Apple's ASC API are DIST=3, DEV‚â•5, MAC_INSTALLER=2 per team (community docs are stale). Each canary run mints 3 throwaway certs, ships full local-mode `fastlane release` to TestFlight, then revokes the 3 just-minted ids on `always()`. Net team-cert delta per run = 0; user's existing shipping certs untouched. Cache-tracked orphan recovery handles partial-failure scenarios. Generator matrix (#126) extends coverage to xcodegen √ó tuist. Cycle stabilization (#127‚Äì#142) closes 17 follow-up bugs surfaced by running the canary at-scale: kwargs forwarding, cert option encoding, build-numbering, keychain partition lists, `.bootstrap.env` synthesis, and the multi-PR TestFlight BBL annotation saga that uncovered a long-standing pilot 2.233.1 bug (`set_changelog` logs success but never POSTs the localization for new builds).

### Added

- `.github/workflows/canary-local-mode.yml` ‚Äî weekly canary workflow (workflow_dispatch + commented-out `cron: '30 11 * * 6'` for forks to opt into; smoketest fork enables it). ## Versioning

This CHANGELOG tracks **template versions** ‚Äî releases of the `apple-shipkit` repo itself, not the forker apps that consume it. Forker apps maintain their own CHANGELOG and version policy independently.

Pre-1.0 (`0.x`) is the development phase: breaking changes are allowed without bumping a major version. The `0.x` retrospective entry above summarizes what shipped during M1‚ÄìM4 milestones; further pre-1.0 work lands under `[Unreleased]` until the public flip.

`v1.0.0` was cut at the public-visibility flip (2026-05-01). From `v1.0.0` onward this project follows strict [Semantic Versioning](https://semver.org/spec/v2.0.0.html): `MAJOR.MINOR.PATCH` where MAJOR is incremented for breaking forker-facing changes (renamed `bin/` scripts, removed Make targets, restructured README sections that forkers cite), MINOR for new forker-facing features (new scripts, new sections), and PATCH for fixes that don't change the forker contract.

Forkers tracking template upgrades should read the CHANGELOG before pulling template changes into their fork. Template-fork creation always uses the default branch (`main`); to base a fork on a specific template version, use `git fetch` to pull a specific tag and rebase manually.

## Tagging

When cutting a release:

1. Move the `[Unreleased]` entries into a new `[X.Y.Z] - YYYY-MM-DD` section in `CHANGELOG.md` and commit.
2. Extract the new section to a release-notes file: `awk '/^## \[X\.Y\.Z\]/{flag=1; print; next} flag && /^## /{exit} flag' CHANGELOG.md > release-notes.md`.
3. Create an annotated tag with the release notes as its message, push, and create the GitHub release.

```bash
# After moving Unreleased ‚Üí [X.Y.Z] in CHANGELOG.md, committing,
# and writing release-notes.md (the new section's body):
git tag -a vX.Y.Z --cleanup=verbatim -F release-notes.md   # annotated tag; --cleanup=verbatim preserves Markdown headings (default --cleanup=strip eats #-prefixed lines); -F reads notes from file
git push origin vX.Y.Z
gh release create vX.Y.Z --notes-from-tag --title "vX.Y.Z" --verify-tag
```
